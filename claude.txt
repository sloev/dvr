Read the whole pipeline — both ideas slot in cleanly, and one's cheaper than you'd think. Here's my assessment grounded in the actual code.

1. Proper audio meters — yes, high value, cheaper than estimated
The groundwork is already here: level name=alevel interval=50000000 (50 ms updates) on the audio tee, and _on_element already pulls the decay array (pipeline.py:309-314). The level element exposes three arrays per channel — peak (instantaneous), rms (energy), decay (held/falling peak) — and has built-in peak-ttl + peak-falloff properties. That means PPM-style peak-hold ballistics are basically free (set hold time + falloff dB/s on the element), not hand-rolled. So:

dBFS scale — level already reports dBFS; our -60…0 range is correct.
PPM ballistics — set peak-ttl (~1.5 s hold) + peak-falloff (~8.6 dB/s ≈ IEC Type II) and read decay. Fast attack is inherent.
VU ballistics — read rms, apply a one-pole ~300 ms integrator in the callback (a few lines).
Clip indicator w/ red hold — latch when peak ≥ -0.1 dBFS, hold ~3 s (or until next clip).
Selectable — a Meter class + a Settings toggle (PPM⇄VU), scale ticks at 0/-6/-12/-20/-40.
Code touch: pipeline.py emits a richer on_level(peak_l, peak_r, rms_l, rms_r); app.py gets a Meter canvas class replacing _draw_vu. Realistically ~150–250 lines, not "few hundred+". 2B cost: ~zero — level is already running. This genuinely matters for borderline Hi8 audio, so I'd build it.

2. Frame grab during record — yes, low cost, with a 2B caveat
Our vtee already has allow-not-linked=true and we have battle-tested add/remove-branch + EOS machinery (_start_clip/_finalize_clip). A still grab is a tiny version of that:

Pipeline.grab_still(path): request a vtee pad, add a transient bin queue leaky=downstream max-size-buffers=1 ! videoconvert ! jpegenc quality=90 ! filesink, probe for the first buffer → inject EOS → tear down. Reuses the exact EOS pattern.
2B caveat: use software videoconvert for the one frame (not the v4l2convert ISP) so a grab during recording doesn't contend with the encoder's ISP context. One 1080p frame through software convert+JPEG is cheap; continuous would not be — so strictly on-demand, never a standing branch.
JPEG, not PNG — smaller, faster, right for scene-logging stills. Saved to <usb>/stills/grab_YYYYmmdd_HHMMSS.jpg.
Works whether or not recording (independent tee branch).
Code touch: ~40–60 lines in pipeline.py + one UI button. Low risk.
